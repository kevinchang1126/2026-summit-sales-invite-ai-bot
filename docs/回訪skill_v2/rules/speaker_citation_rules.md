# 講師引用規則

本文件定義如何依客戶訊號挑選講師、引用數量限制、鼎新優先邏輯。

## 核心原則

1. **鼎新數智講師優先**：李義訓、黃昱凱、黃盈彰
2. **外部講師補強**：林大馗、洪春暉、詹文男、朱浩
3. **引用 2 位以上時，鼎新至少 1 位且排首位**
4. **依聯繫方式限制引用數量**

### Step 2：依聯繫方式決定引用數量

| 聯繫方式 | SPEAKERS 區塊列出 | CONTENT 本文實際提名 |
| --- | --- | --- |
| 電話（phone） | 1-2 位 | ≤ 1 位 |
| 親訪（visit） | 2-3 位 | ≤ 2 位 |
| LINE（line） | 1 位 | ≤ 1 位 |
| EMAIL（email） | 2-3 位 | ≤ 2 位 |

**說明**：SPEAKERS 區塊是給業務參考用（平台可能顯示成「延伸閱讀」），CONTENT 是客戶實際聽到或看到的話術。CONTENT 提名數量永遠 ≤ SPEAKERS 數量。

---

### Step 3：依 tier 調整講師組合

| tier | 組合策略 |
| --- | --- |
| P1 | 2-3 位：以鼎新為主（至少 2 位），補強用外部講師指向治理或場景深度 |
| P2 | 1-2 位：鼎新 1 位 + 外部 0-1 位，避免資訊過載 |
| P3 | 1-2 位：鼎新 1 位為主；可搭 1 位案例型外部講師（詹文男、黃盈彰、朱浩）|
| P4 | 最多 1 位：不強求鼎新優先（因客戶對品牌熟悉度低）|

---

### Step 4：組合檢查

最終選出的 1-3 位講師，組合須滿足：

1. ✅ 若選 2 位以上，鼎新至少 1 位
2. ✅ 若選 2 位以上，鼎新排在 SPEAKERS 陣列首位
3. ✅ 不得重複引用同一位
4. ✅ 不得超過聯繫方式上限（見 Step 2 表）

## 挑選範例

### 範例 A：製造版 P1 客戶、電話

輸入訊號：`Q1_ARRANGE, Q5_SUPPLY_CHAIN, Q7_DATA`
聯繫方式：phone

**挑選過程**：
- Q5_SUPPLY_CHAIN 優先鼎新候選 → 黃盈彰
- Q7_DATA 優先鼎新候選 → 李義訓 或 黃盈彰；外部 → 林大馗
- 聯繫方式電話，SPEAKERS 1-2 位，CONTENT 實際提名 ≤ 1 位
- tier P1，組合策略 2-3 位

**決定**：SPEAKERS 放 2 位（黃盈彰、李義訓），CONTENT 主要提名 1 位（黃盈彰，因供應鏈是主場景）。

**輸出**：
```
## SPEAKERS
- 黃盈彰總經理｜急單跨部門協作與供應鏈 Agent｜K-HYZ-05
- 李義訓副總裁｜企業 AI 治理與可追溯｜K-LYH-03
```

### 範例 B：流通版 P2 客戶、EMAIL

輸入訊號：`Q4_REVENUE, Q5_ONE_YEAR`
聯繫方式：email

**挑選過程**：
- Q4_REVENUE 優先鼎新候選 → 李義訓 或 黃昱凱
- Q5_ONE_YEAR → 三位皆可 + 朱浩
- 聯繫方式 EMAIL，SPEAKERS 2-3 位，CONTENT 提名 ≤ 2 位
- tier P2，組合策略 1-2 位避免過載

**決定**：SPEAKERS 放 2 位（黃昱凱、朱浩），CONTENT 提名 1 位為主。

**輸出**：

輸入：`attendance=no_show`，signals 為空或含 `Q1_NOT_NOW`

**挑選過程**：
- P4 組合策略：最多 1 位，不強求鼎新

```

```

## 邊界情況

### 訊號數量極少（只有 1 個訊號）

- 仍依該訊號查對應表，選 1-2 位鼎新講師
- tier 若為 P1，強制至少 SPEAKERS 2 位以維持推進力道

### 訊號橫跨多題且互相矛盾

例如 `Q1_ARRANGE + Q7_ROI`（高商機 + ROI 疑慮）

- 優先選能同時回應兩者的講師
- 本例：鼎新選黃盈彰（可場景），補強選林大馗（可 ROI）

### 無有效訊號（BEHAVIOR_UNKNOWN）

- 依產業別選一位最「通用」的講師
- 製造預設：黃昱凱（涵蓋最廣）
- 流通預設：李義訓 + 朱浩（流通場景最貼近）

## 引用標記規則（cite 標記）

### 標記語法

在 CONTENT 區塊引用講師觀點時，使用：

```html
<cite code="知識卡代碼">引言文字</cite>
```

### 屬性說明

- `code`：必填，必須是本文件下方「知識卡代碼清單」中列出的有效代碼
- 引言文字：必填，內容見下方長度規則

### 引言長度規則（依聯繫方式）

| 聯繫方式 | 引言長度 | 每則 CONTENT 標記數上限 |
| --- | --- | --- |
| 電話（phone） | 10-20 字 | ≤ 2 個 |
| LINE（line） | 10-20 字 | ≤ 2 個 |
| 親訪（visit） | 30-50 字 | ≤ 4 個 |
| EMAIL（email） | 30-50 字 | ≤ 4 個 |

### 使用範例

**電話（短引言）**：
```
黃盈彰總經理的作法是<cite code="K-HYZ-05">同步看身份、權限、追溯</cite>。
```

**EMAIL（中等引言）**：
```
黃盈彰總經理提到，像供應鏈這類要進正式流程的場景，<cite code="K-HYZ-05">鼎新的作法是同步看三件事——AI 能看什麼、做到哪、怎麼追溯</cite>。
```

### 使用時機

以下情境**應該**使用 `<cite>` 標記：

1. CONTENT 中引用講師的具體觀點、句型、論述
2. CONTENT 中使用鼎新核心語言（代理治理、可控可用可負責）時
3. CONTENT 中提及講師的場景案例（如急單、數智分身、智慧底座）

以下情境**不應該**使用 `<cite>` 標記：

1. 講師稱謂本身（「黃盈彰總經理提到...」這段不標）
2. 客戶輪廓描述語言（「這類規模的製造業」等）
3. 業務自己的話（開場稱呼、收尾 CTA 等）

### 引言文字撰寫原則

- **必須是 CONTENT 話術中自然會出現的文字**，不是額外加的廣告詞
- **不抄錄整段知識卡**，只抓能呈現觀點精髓的一句
- **避免生硬名詞堆砌**，保持口語流暢度
- 親訪/EMAIL 的較長引言，應是完整可理解的半句或一句

### 常見錯誤

❌ **代碼不存在**：
```
<cite code="K-XYZ-99">...</cite>
```
（`K-XYZ-99` 不在知識卡清單中）

❌ **引言過長**（電話場景）：
```
<cite code="K-HYZ-05">鼎新的作法是把 AI 當成有身份的運作單位來管理，不是一個模糊工具，要同步看身份、權限、追溯</cite>
```
（超過 20 字上限）

❌ **引言語氣斷裂**：
```
黃盈彰總經理<cite code="K-HYZ-05">身份權限追溯</cite>。
```
（名詞堆砌，不是自然話術）

✅ **正確寫法**：
```
黃盈彰總經理的作法是<cite code="K-HYZ-05">同步看身份、權限、追溯</cite>。
```
